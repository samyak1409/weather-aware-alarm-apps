# weather-aware-alarm-apps monorepo

Two Flutter alarm apps sharing one core package. **Read SPEC.md before changing any behavior — every product decision there is locked with Samyak; don't silently re-decide.**

## Layout

- `packages/core` — pure logic: solar math, sleep plan, wind engine, Open-Meteo client, theme, models, repos. **All business logic lives here, fully unit-tested.** Apps stay thin.
- `apps/arunoday` — dawn alarm app (daily driver, priority app).
- `apps/nivaat` — wind-conditional badminton alarm.

## Conventions

- UI: pure black `#000000` scaffolds, dark-only, minimal. Accents: Arunoday `AppPalette.dawn`, Nivaat `AppPalette.wind`. No third-party UI kits.
- **Design pass (2026-07-20, Samyak's premium-feel round)**: (1) hero type — ships THIN (the original look); the Settings toggle "Bold clocks & titles" (`Appearance.heavyType`, persisted, default OFF) switches `buildOledTheme(heavyType:)` to w700/w600 heroes with tabular figures; body/labels stay w400 in BOTH modes (the weight contrast IS the look — don't bolden the quiet styles; locked by `design_test.dart`). Each app's `MaterialApp` rebuilds live via `ValueListenableBuilder` on `Appearance.heavyType`. (2) `CraftedBy` (core) renders `CRAFTED WITH ♥ BY SAMYAK` at the foot of BOTH home screens — the heart is `Icons.favorite` (NOT a text glyph: Android font fallback promotes text hearts to the baked-red emoji — device-caught), tapping SAMYAK opens `craftedBySiteUrl`. (3) All motion runs 20% slower: `applyMotionPacing()` (core `motion.dart`, `kMotionSlowdown = 1.2` via `timeDilation` — verified release-safe in Flutter source, it's a plain runtime scale) called at the top of both mains; experiment, revert = delete those two calls. Both Info.plists already had `CADisableMinimumFrameDurationOnPhone` for 120Hz ProMotion. (4) **Launch screens are black** on both OSes (Android `launch_background*.xml` + `Theme.Black` styles, iOS storyboard bg) — never reintroduce the template white.
- **Nivaat navigation (2026-07-20)**: the home top bar has ONE entry point — the settings (tune) icon; courts, alarm sound, and history are settings tiles (that order), and the home last-outcome line is a tap-shortcut to the history sheet. Don't re-add top-bar icons. Arunoday's settings rows are grouped by ritual: wake pair, bedtime group, shared sound, then APPEARANCE, then LOCATIONS. Both settings pages scroll as ONE surface (2026-07-20) through core's `FlashingScrollbar` (the flash-on-open-if-overflowing cue) — no inner scroll regions.
- **App icons (shipped 2026-07-20, switchable)**: all three candidates per app ship; Settings picker drives core's `app_icon.dart` over the `core/app_icon` MethodChannel (ids "1"/"2"/"3"; defaults Horizon/Shuttle). The picker ring updates optimistically — iOS runs the set-icon completion only after its system alert is dismissed — and reverts if the platform refuses. Sources: `branding/` + `tools/make_icons.py` (numpy SDF renderer); `--apply` rewrites the real launcher assets (Android adaptive mipmaps, iOS appiconsets, Settings thumbnails) — never hand-edit those PNGs. Android: MainActivity keeps the MAIN/LAUNCHER intent-filter + default icon (moving it onto aliases breaks `flutter run` after a clean — the tool's pre-build manifest parse only reads real `<activity>` tags); `.IconTwo`/`.IconThree` aliases (manifest-disabled) carry the alternates; switching flips component-enabled states with `DONT_KILL_APP`, safe because the `alarm` + `flutter_local_notifications` plugins resolve entry points via `getLaunchIntentForPackage` (verified in their sources). iOS: `AppIconTwo`/`AppIconThree` appiconsets + `ASSETCATALOG_COMPILER_ALTERNATE_APPICON_NAMES` in every build config; the OS confirms switches with its own alert.
- Alarm scheduling goes through `core`'s `AlarmScheduler` interface, chosen by `createAlarmScheduler`: **`AlarmKitScheduler` on iOS (min target 26.0)**, `AlarmPkgScheduler` on Android. **iOS is AlarmKit-only — there is NO `alarm`-package fallback** (2026-07-18): if the user denies AlarmKit, `scheduleRing` silently no-ops (`PlatformException` caught) and `AlarmPermissionBanner` (shown on both home screens, driven by `alarmSchedulingDenied()`) nudges them to Settings via `url_launcher` (`app-settings:`). So `AlarmPkgScheduler` is effectively Android-only (`warningNotificationOnKill: false` — Android's ring genuinely survives kill). **Both wake and bedtime rings use AlarmKit on iOS** (2026-07-16: a real system alarm that survives force-quit/reboot beats the `alarm` package's in-process Timer; cost — Arunoday's bedtime "+1h" ritual is **Android-only**, since AlarmKit alerts are Stop-only). Never call the `alarm` package directly from screens, except ring-screen `Alarm.ringing` listeners. (AlarmKit's auth prompt is simulator-verified; real-device iOS ring behavior is still untested.)
- **Cross-platform parity is mandatory — every feature must work on BOTH Android and iOS.** Never ship a path that works on one and silently no-ops on the other. When touching anything platform-shaped (notifications, permissions, plugins, `dart:io`, `Platform.is*`), explicitly verify the _other_ platform has a working equivalent — don't assume a plugin handles it. **Burned once (2026-07-13):** we assumed "the `alarm` package requests iOS notification permission" — it does NOT (it only _checks_ `authorizationStatus` and silently drops the notification), so both apps requested permission on Android only and Nivaat's skip cards / Arunoday's bedtime banner would never have appeared on iOS. Fix: both notifiers called `IOSFlutterLocalNotificationsPlugin.requestPermissions` too. (2026-07-20: Arunoday's iOS request was then removed as vestigial — under AlarmKit-only iOS it posts zero notifications, so parity here means *nothing to grant*; Nivaat still requests on both.) iOS permission is a single app-level grant, so requesting from multiple places prompts only once — the "avoid double-prompt" fear was unfounded. Rule: **read the plugin's native source before trusting it to request a permission.**
- **Permission nudges (2026-07-19)**: a denied permission always leaves a persistent home-screen banner with a one-tap fix — `NotificationPermissionBanner` (both apps; Arunoday Android-only, and since 2026-07-20 Arunoday doesn't request the permission on iOS at all — it posts zero iOS notifications under AlarmKit-only iOS), `BackgroundChecksBanner` (Nivaat: Android battery exemption / iOS Background App Refresh), `AlarmPermissionBanner` (iOS AlarmKit). All render through core's `NudgeBanner` shell and re-check on app resume. No banner ever flashes behind its own first-run dialog: the notification banners gate on a per-app "asked" flag + `recheckAfter` (the startup request future), the battery banner on `batteryAskInFlight` (set around the once-ask, cleared by the resume that answers it).
- **Nivaat history (2026-07-19, append-only since 2026-07-20)**: history is an event log — the heads-up snapshot row (written at T with `watchedUntil` = the +30m cap, never touched by retries) and the final outcome row (cap skip, or late ring's "rang") are SEPARATE permanent entries; nothing overwrites. `NivaatStore.upsertHistory` (keyed `alarmId + at + kind`) exists only to converge racing double-writes of the same event. An audible ring is logged mid-ring (Rule 1); every state-discard path (edit/toggle/delete → `engine.discardOccurrence`, never a blind `clearCheckState`) finalises a fired ring first; finalisation rolls straight into evaluating the next occurrence. Foreground resyncs call `NivaatStore.refresh()` (SharedPreferences reload) first — per-isolate caching otherwise hides background-check writes until a cold start.
- Times: all solar math is UTC-internal, converted at the edge. Location = saved lat/lon, never GPS.
- **Whole minutes only in the app layer**: `core` computes dawn/sunrise to the second, but the apps floor them to the minute at the controller boundary (`dawnOn`/`sunriseOn`). Dawn's seconds drift ~±25s/day; letting them into wake times makes every derived display (picker anchors, ring moments, sleep math) flicker by one minute. Never quote a second-precise time to the user.
- Tests: `packages/core/test` validates against real-world reference numbers recorded in SPEC.md (Jaipur/Tonk/BLR dawn times, ring-rate expectations). Keep them passing; they are the spec-as-code.
- Verify with `flutter analyze` + `flutter test` in each package/app you touch.
- **Coverage (2026-07-21): 169 tests (core 107, arunoday 27, nivaat 35).** Business logic ~95% line-covered (core 58.0% overall but format 100%, repos 99%, models/wind 96%, app_icon/sleep_plan 95-96%, solar 98%, nudge_banner/motion/flashing_scrollbar 100%, notification_permission_banner 91%, appearance 89%, crafted_by 83%; arunoday controller 100%; nivaat controller 100%, engine 95.4%). Remaining new-module misses are unreachable-by-design (private ctors), const-ctor lcov artifacts, or plugin wrappers (crafted_by's url_launcher call) — same policy as the wrapper list below. The uncovered remainder is deliberately not unit-tested: platform-plugin wrappers (`alarm_pkg_scheduler`, `alarmkit_scheduler`, `skip_notifier`, `notifications`, `battery_optimization`, `system_settings`), the `open_meteo` HTTP client, and Flutter widgets (`ring_gate`, `sound_picker`, `alarm_permission_banner`, `background_banner`, screens) — these need the device/simulator smoke tests and an integration-test screenshot pass, not unit tests. (`scheduler_test` covers the pure non-iOS guards of `createAlarmScheduler`/`alarmSchedulingDenied` plus `NoOpAlarmScheduler`; `check_scheduler_test` covers Android init soft-fail + Error propagation; `home_note_test` locks N10 soft-wrap / no hard `\n`; `location_picker_test` covers the GPS name dialog's fast-Save lifecycle and geocode Exception→message / Error→rethrow via `reportLocationSearchFailure`.)

## Build notes (hard-won, don't rediscover)

- JDK: brew formula openjdk@21, wired via `flutter config --jdk-dir`. Android SDK: `~/Library/Android/sdk` (headless, no Android Studio); use the SDK's own `cmdline-tools/latest/bin/avdmanager` (the brew one points at the wrong root).
- Both apps force `compileSdk = 36` AND carry a root-gradle reflection override forcing plugin modules to 36 (the `alarm` plugin pins 34 but its `flutter_fgbg` dep needs 35+).
- Nivaat Android needs core-library desugaring (flutter_local_notifications).
- **Nivaat release R8 (2026-07-20):** Flutter enables minify/shrink on release APKs. Without `apps/nivaat/android/app/proguard-rules.pro`, R8 horizontally merged `android_alarm_manager_plus`'s `FlutterBackgroundExecutor` into `androidx.lifecycle.ServiceLifecycleDispatcher` → release APK “Nivaat keeps stopping” on first tap (`AndroidAlarmManager.initialize()` in `main` before `runApp`). Debug builds don't minify, so `flutter run` never caught it; Arunoday is unaffected (no that plugin). `AndroidCheckScheduler.initialize` also soft-fails so a future native glitch can't brick launch. **`AndroidAlarmManager.initialize()` is deferred to after the first frame** (still before cascade booking) so its background FlutterEngine can't race the main engine during cold start. **Also keep `androidx.work.**` + Room** — Nivaat alone pulls `workmanager`; R8 stripping `WorkDatabase_Impl` killed release installs at `InitializationProvider` before Flutter ran (`NoSuchMethodException`), while `flutter run` debug looked fine. After editing those keep rules: `flutter build apk --release` and check mapping still has `FlutterBackgroundExecutor -> FlutterBackgroundExecutor` and `WorkDatabase_Impl`.
- **Launch soft-fail + iOS checks.initialize (2026-07-21):** Both apps wrap `scheduler.ensureInitialized()` in `on Exception` (never brick cold start on a plugin hiccup; programming `Error`s still propagate). Nivaat's `resync` and `CheckScheduler.initialize` use the same Exception-only policy; location search geocode goes through `reportLocationSearchFailure` (inline “Search failed — check network” for Exceptions; Errors rethrown with stack). Startup goes through `_finishStartup`: **`checks.initialize()` on BOTH platforms** (iOS seeds the periodic BGAppRefresh Dart submit — AppDelegate only registers the handler; Android still defers past the first frame for the AlarmManager race) then `controller.init()`. Don't drop the iOS call when touching Android deferral — burned once in `d7ba843`.
- **AGP is 9.0.1 — plugins that self-apply KGP only under AGP < 9 silently fail to compile their Kotlin (class-not-found in `GeneratedPluginRegistrant`).** Burned by `app_settings` 8.0.0 (2026-07-18): it skips `kotlin-android` on AGP≥9, so `AppSettingsPlugin` never built → both apps' Android builds broke. Fix: dropped it for `url_launcher` (Flutter-team, KGP-clean) for the iOS "Open Settings" (`app-settings:`). **Lesson: after any plugin/native change, `flutter build apk` — `flutter test` does NOT build the Android/iOS native side, so plugin-compile breaks stay invisible.**
- Nivaat Android battery-optimization exemption: `MainActivity` exposes a `nivaat/battery` MethodChannel (`isExempt`/`requestExempt` → `ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`); `requestBatteryExemptionOnce()` (in `main`) asks once so off-charger Doze doesn't throttle the cascade's network checks. Needs `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` in the manifest. **`nivaat/battery` is two-sided (2026-07-19): the iOS AppDelegate answers `isExempt` too — there it means "Background App Refresh available" — feeding the same `BackgroundChecksBanner`.**
- **`core/system_settings` MethodChannel (2026-07-19): implemented in BOTH MainActivities** (keep them in sync) — serves core's `openNotificationSettings()` (`ACTION_APP_NOTIFICATION_SETTINGS`; Android's notification page is unreachable by URL, and `permission_handler`/`app_settings`-style plugins are an AGP-9 risk — see below). iOS needs no channel (`app-settings:` URL).
- **`core/app_icon` MethodChannel (2026-07-20): implemented in BOTH MainActivities AND BOTH AppDelegates** (keep all four in sync) — `get`/`set` with icon ids "1"/"2"/"3" (see the app-icons convention bullet). Dev caveat: after switching to a non-default icon on a device, `flutter run`'s explicit `am start …/.MainActivity` fails (component disabled as launcher); fix with `adb shell pm enable com.samyak.<app>/com.samyak.<app>.MainActivity` or switch back to icon 1 in Settings — documented in RUNNING.md troubleshooting.
- Min OS: **Android minSdk 33** (Android 13; set in each `app/build.gradle.kts`) · **iOS deployment target 26.0** (AlarmKit-only ring path; set in each `Runner.xcodeproj/project.pbxproj` and Nivaat's `Podfile`). Nivaat also uses workmanager: the Swift import is `workmanager_apple` (federated module name), plugin class `WorkmanagerPlugin`. **Two iOS BGTask ids** in Info.plist `BGTaskSchedulerPermittedIdentifiers` + registered in `AppDelegate`: `…refresh` (periodic BGAppRefresh, `registerPeriodicTask`) and `…processing` (BGProcessing, `registerBGProcessingTask`). workmanager's `registerProcessingTask(initialDelay:)` **does** set the iOS `earliestBeginDate` (= now+delay); `Constraints.requiresCharging`→`requiresExternalPower`, `networkType:.connected`→`requiresNetworkConnectivity`. BGProcessing is one-shot (no native re-submit) so `scheduleCheck` re-registers it each `evaluateAll`. (`registerOneOffTask`'s `initialDelay` is genuinely ignored on iOS — don't use it for scheduling.)
- AlarmKit: `flutter_alarmkit` takes Flutter asset paths for sounds (WAV < 30s); timestamps are Unix **milliseconds** as double; it assigns its own UUIDs → `AlarmKitScheduler` persists an int-id→UUID map. `NSAlarmKitUsageDescription` required in Info.plist.
- Emulator smoke loop: boot `emulator -avd pixel -no-window`, `adb exec-out screencap -p`. iOS: `xcrun simctl` install/launch/screenshot; seed SharedPreferences by writing the app container's plist while the sim is shut down (simctl spawn defaults writes outside the sandbox — doesn't work). **Both platforms store Flutter prefs under the `flutter.` key prefix** (`flutter.arunoday.settings`, etc.).

- `flutter run` prints a KGP deprecation warning ("plugins that apply Kotlin Gradle Plugin (KGP): alarm") — benign, lives in the `alarm` plugin's own build script, nothing on our side to change; goes away when the plugin migrates to Flutter's built-in Kotlin. Track upstream releases.
- **Never run `dart format` over whole directories (2026-07-20)**: the current SDK ships the new *tall-style* formatter, and the repo predates it — a blanket format rewrote 33 files including untouched ones, drowning the real diff. Match the surrounding (old-style) formatting by hand; if a repo-wide reformat is ever wanted, it must be its own dedicated commit Samyak asks for.

## Assets & screenshots

- `screenshots/<app>/` is **git-tracked** with stable filenames — history lives in git, never in versioned folder names. Exactly **3 shots per app × OS**: `01_empty_state`, `02_home`, `03_settings`. Refresh them **only when Samyak explicitly asks** ("update screenshots"), never automatically after UI changes. Capture **one OS at a time** (never leave Pixel emulator + iOS Simulator both open). Optional aid: `--dart-define=SCREENSHOT_HARNESS=true` (see `screenshots/README.md`); if a system dialog blocks a shot, **ask Samyak to tap** instead of fighting automation.
- Alarm sounds are generated, not sourced: `tools/make_sounds.py` (numpy) synthesizes both WAVs; Nivaat's loudness variants (75-100%, 5% steps) are amplitude-scaled copies of the base. Keep every sound < 30s (AlarmKit hard limit).
